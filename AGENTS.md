# AGENTS.md

## 项目概述

AI驱动的测试用例生成系统，基于"智能三模块架构"。系统通过 AI 交互和工具脚本，将 Word 产品文档转换为专业的 Excel 测试用例文件，支持大文档处理（3000+ 行）。

## 常用命令

```bash
# 安装依赖
pip install -r requirements.txt

# 安装 Pandoc（Word 转换必需）
# macOS: brew install pandoc
# Linux: sudo apt-get install pandoc

# 阶段1：Word 文档转 Markdown
python script/convert_docx_to_markdown.py

# 阶段5：JSON 合并生成 Excel
python script/generate_final_excel.py

# 可选工具
python tools/data_validator.py
python tools/project_manager.py status
python tools/project_init.py my_project
```

## 目录结构

```
├── script/                    # 核心脚本
│   ├── convert_docx_to_markdown.py   # Word→Markdown 转换器
│   └── generate_final_excel.py       # JSON→Excel 生成器
├── product/                   # 产品文档
│   ├── *.docx                # 原始 Word 文档（放置于此）
│   ├── markdown/             # 转换后的 Markdown 文件
│   └── modules/              # 分割的模块文件（大文档用）
├── output/                    # 生成的测试用例和报告
│   ├── module_*.json         # 按模块分的测试用例 JSON 文件
│   └── *_测试用例_*.xlsx      # 最终 Excel 测试用例文件
├── tools/                     # 可选工具
│   ├── data_validator.py     # 数据验证工具
│   ├── project_manager.py    # 项目管理工具
│   └── project_init.py       # 项目初始化工具
├── config.py                 # 配置管理
└── AGENTS.md                 # 本文件
```

## 关键配置

通过 `config.py` 管理：
- **文档分割**：超过 2000 行触发智能分割
- **测试维度**：12 个综合测试维度
- **Excel 格式**：13 列专业结构，智能单元格合并
- **文件模式**：`module_*.json`（新格式），`batch_*.json`（旧格式）

# 智能三模块架构规则体系

以下规则在执行测试用例生成任务时自动生效，指导 AI 完成完整的三模块工作流。

---

## 一、自动执行触发与流程

### 触发条件

当用户上传产品文档（.docx 文件）或提供文档路径，并表达生成测试用例的需求时，AI 必须立即自动启动完整的三模块流程。

### 强制连续执行规则

**AI 必须在一次回复中完成所有步骤，不得中断：**
- ❌ 禁止：完成第一批次后说"自动继续第二批次"然后停止
- ❌ 禁止：只生成部分批次就结束回复
- ❌ 禁止：没有调用 Excel 生成工具就结束
- ✅ 必须：连续执行所有批次直到生成最终 Excel 文件
- ✅ 必须：提供完整的执行报告和文件位置

### 自动执行序列

```
步骤1：文档检测和转换
├── 自动扫描 product 目录查找 .docx 文件
├── 自动调用：python script/convert_docx_to_markdown.py
└── 确认生成 MD 文件

步骤2：启动产品分析师模块（强制扫描机制）
├── 【强制步骤2.1】执行完整扫描命令（必须执行，不可跳过）
│   ├── 执行：grep -n "^[0-9]\+\." {文档路径}
│   ├── 执行：grep -n "^[0-9]\+\.[0-9]\+" {文档路径}
│   ├── 执行：grep -n "^[0-9]\+\.[0-9]\+\.[0-9]\+" {文档路径}
│   ├── 执行：grep -n "\*\*.*弹窗\|页面\|模块.*\*\*" {文档路径}
│   └── 输出：【模块完整性扫描报告】
│
├── 【强制步骤2.2】统计三级章节内容行数（必须计算）
│   ├── 根据 grep 输出的行号，计算每个三级章节的内容行数
│   ├── 公式：下一章节行号 - 当前章节行号 = 内容行数
│   └── 输出：【三级章节行数统计表】
│
├── 【强制步骤2.3】创建完整映射表（基于真实扫描结果，不可跳过）
│   ├── 列出所有扫描到的章节（使用 grep 扫描到的实际名称）
│   ├── 为每个章节评估独立性（基于行数>80或复杂度高）
│   ├── 确定是否需要独立测试模块
│   └── 输出：【章节→模块映射表】（章节名→模块名→JSON文件名）
│
├── 【强制步骤2.4】验证映射完整性（数量对比，不可跳过）
│   ├── 对比扫描结果总数和映射表数量
│   ├── 检查是否有遗漏的章节（特别是三级章节）
│   └── 输出：【预计生成X个模块】（X必须是具体数字）
│
└── 【强制步骤2.5】锁定模块列表
    └── 输出最终的 module_list（JSON格式）供测试生成师使用

步骤3：启动测试用例生成师模块（测试场景强制识别）
├── 基于产品分析师的模块列表进行测试生成
├── 【强制步骤3.1】深度分析模块文档（生成前必须执行）
│   ├── 读取该模块对应的文档章节内容（start_line到end_line）
│   ├── 识别所有功能点、交互点、数据点、状态点
│   ├── 提取业务规则、验证规则、异常处理逻辑
│   └── 输出：【模块XX测试场景清单】
│
├── 【强制步骤3.2】验证场景覆盖12个维度（不可跳过）
│   ├── 功能维度：正常流程场景2-4个
│   ├── UI交互维度：按钮/弹窗/跳转场景2-3个
│   ├── 数据验证维度：字段/格式/长度场景2-3个
│   ├── 异常处理维度：网络/错误/失败场景2个
│   ├── 边界值维度：最大/最小/临界场景1-2个
│   ├── 兼容性维度：安卓/iOS/屏幕场景1-2个
│   └── 输出：【场景覆盖检查】标注是否全面
│
├── 【强制步骤3.3】场景不足自动补充（不可跳过）
│   ├── 如果场景清单<11个 → 重新深度分析文档
│   ├── 检查哪些测试维度未覆盖
│   └── 禁止跳过，直到场景≥11个
│
├── 【强制步骤3.4】基于场景清单生成测试用例（一场景一用例）
│   ├── 每个场景生成1个详细的测试用例
│   ├── 用例必须基于文档真实内容，不编造
│   ├── 用例步骤具体可执行，不使用"xxx"占位符
│   └── 输出：module_XX.json
│
├── 【自检】生成后自我检查
│   ├── 统计用例数
│   ├── 如果<11个 → 场景遗漏，重新分析文档
│   ├── 如果>14个 → 检查是否有重复场景
│   └── 输出：【模块XX：Y个用例，覆盖Z个场景】
│
├── 【禁止行为】
│   ├── ❌ 禁止为了凑数量而生成无意义用例
│   ├── ❌ 禁止不分析文档直接快速生成
│   ├── ❌ 禁止跳过场景清单直接生成用例
│   └── ❌ 禁止场景描述泛泛（如"功能测试"）
│
├── 为每个用例提取需求原文（用于追溯）
├── 【进度检查点】每生成3个模块后输出进度
│   └── 格式：已生成X/预计Y个模块，剩余：[模块列表]
└── 输出所有批次JSON文件

步骤3.5：需求覆盖验证（强制执行）
├── 提取需求文档中所有功能描述句
├── 构建覆盖矩阵：需求点 vs 测试用例
├── 检查覆盖率是否达到100%
├── 如覆盖率<100%，补充遗漏的测试用例
└── 输出覆盖检查报告

步骤4：启动测试用例检查师模块
├── 执行核心原则检查
├── 【最终验证】执行：ls output/module_*.json | wc -l
├── 对比：实际文件数 vs 预计模块数（来自步骤2.4）
├── 输出验证结果：
│   ├── ✅ 预计模块数：X个
│   ├── ✅ 实际生成数：Y个
│   ├── ✅ 验证状态：[通过/失败]
│   └── ✅ 缺失模块：[如有列出]
├── 如不匹配 → 自动补充缺失模块，直到数量匹配
├── 自动合并所有批次JSON文件
├── 自动调用：python script/generate_final_excel.py
└── 输出最终Excel文件

步骤5：完成通知
├── 提供执行摘要和质量报告
├── 显示最终Excel文件位置
└── 确认所有检查通过
```

### 三大 AI 角色

- **产品分析师**（模块1）：文档分析、模块识别、智能分割
- **测试生成师**（模块2）：测试用例创建，12维度覆盖
- **测试检查师**（模块3）：质量验证、报告生成

## 二、核心原则

### 1. 路径规范化
- 不指定具体产品文件名，统一使用 `product/` 路径下的文档
- 不指定具体 md 文件名，统一使用 `product/markdown/` 路径下的文档
- 支持自动扫描和选择文档

### 2. AI 针对性分析
- 每次根据产品文档内容进行深度分析
- 完全基于文档内容生成测试用例，不使用任何硬编码
- 针对不同类型的产品文档采用不同的分析策略

### 3. 测试覆盖全面性
- 严格按照用户定义的测试覆盖规则执行
- 确保无任何测试点遗漏
- 覆盖所有功能、场景、异常、边界等情况

### 4. Excel 格式规范
- 生成的 Excel 文件严格符合格式要求
- 包含所有必需的列和格式要求
- 支持多平台测试结果记录
- 模块排序规范：测试用例按照业务逻辑顺序排列

### 5. 文档顺序强制要求
- AI 必须严格按照文档中模块的出现顺序排列测试用例
- 这是最高优先级的不可违背规则
- 任何其他排序逻辑都不能覆盖文档原始顺序
- 生成的 Excel 文件中模块顺序必须与文档一致

### 6. 测试用例去重强制要求
- 绝对禁止生成重复或高度相似的测试用例
- 每个测试用例的测试步骤必须是独特的
- 严格禁止使用通用模板批量生成测试用例
- 这是不可违背的质量底线要求

## 三、模块完整性识别强制要求

AI 必须执行系统化的三层扫描确保无模块遗漏。任何测试用例生成前都必须完成完整性验证。

### 3.1 三层结构扫描法

#### 第一层：主章节扫描
- 使用正则匹配识别所有一级标题：`^\d+\.`
- 提取所有主要章节，严格按文档出现顺序记录
- 输出格式：章节号. 标题名称 [行号:XXX]

#### 第二层：子模块扫描
- 使用正则匹配识别所有二级标题：`^\d+\.\d+`
- 识别每个主章节下的子模块
- **关键原则**：每个子模块都可能是独立的测试模块
- 输出格式：章节号.子节号 子模块名称 [行号:XXX] [父模块:XXX]

#### 第三层：特殊功能扫描
- 识别粗体标题：`\*\*.*\*\*`
- 识别表格中的功能模块
- 识别弹窗、页面等独立功能：`\*\*.*弹窗.*\*\*`、`\*\*.*页面.*\*\*`
- 识别管理功能：`\*\*.*管理.*\*\*`
- 识别提醒功能：`\*\*.*提醒.*\*\*`
- 输出格式：功能名称 [类型:弹窗/页面/组件] [行号:XXX]

### 3.2 模块独立性判断标准

#### 判断优先级（从高到低）：

**第一优先级：独立UI判断（最高优先级，一票决定）**
- 有独立的用户界面（页面/弹窗） → **必须独立为模块，无论行数多少**
- 此规则不可被行数标准覆盖

**第二优先级：业务流程判断**
- 有独立的业务流程（5个以上测试场景）→ 独立
- 有独立的数据处理逻辑 → 独立
- 与其他功能耦合度低 → 独立
- 文档中有详细的功能描述（超过3段）→ 独立

#### 合并测试条件（必须同时满足所有条件才可合并）：
- **无**独立的用户界面（无独立页面/弹窗）
- 功能简单（少于3个测试场景）
- 与主功能强耦合
- 纯展示或配置类功能

#### 小模块用例数量说明：
- 独立模块的用例数量按实际场景数量生成
- 如果模块行数较少（<50行），实际场景可能少于11个，此时按实际场景数量生成即可
- **禁止为凑数量（11个）而生造与文档无关的测试用例**

### 3.3 三级章节独立性判断标准

**独立UI标准**（优先级最高）：
- 章节描述的是独立页面或弹窗 → 必须独立，不受行数限制

**内容行数标准**（仅用于无独立UI的章节）：
- 行数 > 150行 → 强烈建议独立
- 行数 > 80行 → 倾向独立
- 行数 50-80行 → 根据业务复杂度判断
- 行数 < 50行 → 建议合并到父模块

**业务复杂度标准**：
- 包含3个以上独立操作流程 → 建议独立
- 包含独立的筛选功能 → 建议独立
- 包含状态管理和权限控制 → 建议独立

**关键词识别标准**：
- 章节名包含"模块"、"系统"、"管理" → 建议独立

### 3.4 完整性验证清单

生成模块列表后，必须验证：
- □ 是否覆盖了所有主章节？
- □ 是否覆盖了所有子模块？
- □ 是否遗漏了重要的弹窗？
- □ 是否遗漏了独立的页面？
- □ 模块数量是否合理（建议8-15个）？
- □ 每个识别的功能点都有归属模块？

### 3.5 强制完整性验证（零遗漏机制）

**AI 必须实际执行 grep 命令，禁止编造或凭印象列举。**

必须执行的命令：
```bash
# 一级章节扫描
grep -n "^[0-9]\+\\." product/markdown/*.md

# 二级章节扫描
grep -n "^[0-9]\+\\.[0-9]\+" product/markdown/*.md

# 三级章节扫描
grep -n "^[0-9]\+\\.[0-9]\+\\.[0-9]\+" product/markdown/*.md

# 特殊功能扫描
grep -n "\\*\\*.*弹窗\|页面\|模块.*\\*\\*" product/markdown/*.md
```

#### 验证关键字检查（防止 AI 作弊）

AI 的输出必须包含以下验证关键字，否则视为未实际执行 grep：
1. ✅ "Exit code: 0" 或 "Exit code: 1" 至少出现3次
2. ✅ "grep -n" 至少出现3次
3. ✅ 至少一个真实的章节名（行号:章节内容格式）
4. ✅ "小计：X个" 至少出现3次（X必须是数字）
5. ✅ "预计生成模块总数：X个"（X必须是具体数字）

**如果缺少任何验证关键字 → 视为违规，自动终止并重启流程**

#### 强制执行清单（AI 必须逐项完成）

生成测试用例前必须确认：
- [ ] 已执行 grep 扫描（不是凭印象）
- [ ] 已输出完整扫描报告（包含所有层级）
- [ ] 已统计三级章节行数（用于独立性评估）
- [ ] 已创建章节→模块映射表（每个章节有明确去向）
- [ ] 已明确预计模块数（具体数字，不是"约"）
- [ ] 扫描报告包含所有验证关键字
- [ ] 可以开始生成测试用例

**只有全部打勾才能开始生成！违反=执行失败**

#### 自动补充机制

生成完成后的验证流程：
```bash
# 步骤1：统计实际生成的文件数
ls output/module_*.json | wc -l

# 步骤2：对比预计数和实际数
# 预计：X个（来自扫描报告）
# 实际：Y个

# 步骤3：如果不匹配，自动识别缺失模块并补充

# 步骤4：输出验证结果
# 【最终验证结果】
# ✅ 预计模块数：X个
# ✅ 实际生成数：Y个
# ✅ 验证状态：通过/失败
# ✅ 缺失模块：[如有列出]
```

### 3.6 模块遗漏风险识别

#### 高风险遗漏场景：
- 表格中的功能描述
- 图片说明中的操作流程
- 嵌套弹窗功能
- 条件分支处理逻辑
- 状态转换机制
- 错误处理专门流程
- 权限相关功能
- 后台管理功能

#### 识别策略：
- 关键词扫描：页面、弹窗、操作、功能、模块、组件
- 动词扫描：点击、输入、选择、跳转、显示、处理
- 名词扫描：按钮、列表、表单、卡片、标签、状态

## 四、文档分析与格式识别规则

### 4.1 通用文档格式适配

AI 必须能够智能识别和处理各种文档格式：

#### 标题格式自动识别：
- Markdown标题：# ## ### #### 等任意级别
- 粗体标题：**标题内容** 格式
- 中文序号：一、二、三、四、五 等
- 中文括号序号：一）二）三）等
- 阿拉伯数字序号：1、2、3 等
- 阿拉伯数字括号：1）2）3）等
- 点号格式：1. 2. 3. 等
- 表格中的标题：| **标题** | 格式
- 其他特殊格式：根据上下文智能识别

#### 内容结构智能分析：
- 自动识别章节层级关系
- 理解段落间的逻辑关系
- 识别列表、表格、图片等内容类型
- 提取关键业务信息和功能描述

### 4.2 动态内容提取规则

AI 需要根据文档实际内容动态提取业务信息，不依赖固定模式：

#### 业务功能识别：
- 通过语义分析识别核心业务功能
- 从文档描述中提取操作步骤
- 识别业务流程和数据流向
- 理解业务规则和约束条件

#### 上下文理解：
- 理解段落间的逻辑关系
- 识别前置条件和依赖关系
- 理解业务场景的完整流程
- 考虑异常情况和边界条件

### 4.3 格式清理与标准化规则

#### 自动格式符号识别和清理：
- 移除各种标题前缀：#、*、数字序号、中文序号等
- 清理 Markdown 格式符号：**、__、[]()、![]( 等
- 移除表格格式符号：|、--- 等
- 清理 HTML 标签和 CSS 样式属性
- 移除图片引用和媒体链接

#### 长度和质量控制：
- 智能控制模块名称长度（2-15字符）
- 提取长标题中的核心关键词
- 确保提取结果的语义完整性

### 4.4 文档类型自适应

#### 需求文档：重点提取功能需求和业务规则
#### 技术文档：将技术描述转换为测试场景
#### 操作手册：提取操作步骤和流程
#### 业务流程文档：理解端到端的业务流程

## 五、动态模块识别与标准化规则

### 5.1 文档顺序保持要求（核心要求）

- AI 在识别模块时必须记录每个模块在文档中的首次出现位置
- 最终模块列表必须按照文档中的出现顺序排列
- 不允许因为任何其他逻辑（如字母顺序、重要性等）改变文档原始顺序
- 这是最高优先级的要求，任何其他排序逻辑都不能违背此原则

#### 具体实现：
1. 逐行扫描文档内容，记录每个业务模块标题的出现行号
2. 为每个识别的模块分配一个基于文档位置的序号
3. 按照序号从小到大的顺序排列最终模块列表

#### 排序方式：
- ❌ 按字母顺序
- ❌ 按重要性
- ✅ 按文档顺序：严格按照文档中出现的先后顺序

### 5.2 完全基于文档内容
- 不使用固定的通用模块
- 完全基于产品文档的实际业务内容识别模块
- 每个文档都应生成对应的特定业务模块
- **模块名称必须来源于文档中的实际章节标题**，仅做格式清理，不可改写语义

#### 模块命名禁止行为：
- ❌ 禁止 AI 自创模块名称（如将"提交订单的入口"改为"结算入口"）
- ❌ 禁止拼接多个章节名形成新名称（如"收银台与信用卡"、"单笔支付与订单"）
- ❌ 禁止使用文档中不存在的概括词（如"管理"、"与"、"及"）
- ❌ 禁止因合并而丢弃章节原名

#### 模块命名正确示例：
- 文档标题"提交订单的入口" → 模块名"提交订单入口"（仅移除"的"）✅
- 文档标题"我的地址列表页" → 模块名"地址列表页"（移除"我的"）✅
- 文档标题"💳添加信用卡弹窗" → 模块名"添加信用卡弹窗"（移除emoji）✅
- 文档标题"提交订单页（合单页）" → 模块名"提交订单页"（移除括号补充说明）✅
- ❌ 错误：将"提交订单的入口"+"我的地址列表页"合并命名为"结算入口与地址选择"

### 5.3 模块名称标准化通用规则

#### 格式清理规则：
- 移除标题格式符号：# * emoji等
- 移除序号前缀：1）2）一）二）等
- 移除无意义后缀：_需求 _说明 _详情等
- 移除括号内的补充说明（如"（合单页）"）
- 移除副标题（如" - 支持卖家修改发货地"）

#### 长度控制规则：
- 模块名称长度控制在2-15个字符
- 过短（<2字符）需补充上下文
- 过长（>15字符）需提取核心关键词

#### 语义优化规则：
- 将技术术语转换为业务术语
- 统一同义词表达（如"登陆"→"登录"）
- 简化复合词汇（如"用户账户管理"→"账户管理"）
- **仅允许简化，不允许改写或替换为文档中不存在的词**

### 5.4 文档章节过滤原则

必须准确识别并过滤掉文档管理类章节，避免误判为业务模块：
- **文档管理章节**：修订记录、版本历史、变更日志、文档说明
- **文档结构章节**：目录、概述、背景、引言、需求概述、项目背景、术语定义
- **模式匹配**：包含"文档"、"版本"、"修订"、"记录"、"概述"、"背景"等词汇

### 5.5 章节性质分类（强制执行）

**在模块识别之前，必须先对文档所有一级章节做性质分类。**

| 性质分类 | 处理方式 | 典型章节 |
|---------|---------|---------|
| **文档管理类** | 完全过滤，不读取不生成 | 修订记录、版本历史、变更日志 |
| **背景概述类** | 只读理解上下文，**禁止**作为测试点来源 | 背景、需求来源、需求目标、需求概述、核心流程 |
| **需求描述类** | ✅ **唯一的用例生成来源** | 需求详细描述、功能说明、交互说明 |
| **参考信息类** | 辅助补充到对应功能模块用例中，不独立建模块 | 多语言管理、埋点、设计图链接 |

#### 背景概述类详细说明：
- 包含KR指标、转化率目标、竞品调研、需求来源人、上下游方案链接等
- 这些内容帮助AI理解业务上下文和优先级，但**不是可测试的功能点**
- ❌ 禁止将"转化率提升至60%"等目标指标生成为测试用例
- ❌ 禁止将"电通指出不符合日本用户习惯"等背景描述生成为测试用例

#### 参考信息类详细说明：
- 多语言key、埋点事件等信息**融入**对应功能模块的用例中验证
- 例如：在"提交订单入口"模块的用例中同时验证中日文文案
- 不为"多语言管理"单独创建测试模块

#### 强制输出格式：
```
【章节性质分类表】
章节名称          | 性质分类    | 处理方式
─────────────────┼────────────┼──────────
修订记录          | 文档管理类  | 完全过滤
一、背景          | 背景概述类  | 只读不生成
三、需求详细描述   | 需求描述类  | ✅ 用例来源
四、多语言管理     | 参考信息类  | 融入功能模块
```

## 六、12维度测试覆盖规则

### 6.1 维度定义

| 维度 | 覆盖内容 |
|------|----------|
| 1. 功能覆盖 | 基础功能、核心功能、辅助功能、功能组合 |
| 2. 场景覆盖 | 正常场景、典型场景、复杂场景、端到端场景 |
| 3. 异常覆盖 | 输入异常、状态异常、外部异常、业务异常 |
| 4. 边界覆盖 | 数据边界、时间边界、权限边界、容量边界 |
| 5. 数据覆盖 | 数据类型、数据格式、数据长度、数据精度 |
| 6. 界面覆盖 | 界面元素、界面布局、界面交互、界面反馈 |
| 7. 流程覆盖 | 线性流程、分支流程、循环流程、回退流程 |
| 8. 权限覆盖 | 角色权限、功能权限、数据权限、操作权限 |
| 9. 性能覆盖 | 响应时间、并发性能、数据量性能、资源使用 |
| 10. 兼容性覆盖 | 浏览器兼容、操作系统兼容、设备兼容、版本兼容 |
| 11. 安全覆盖 | 输入安全、认证安全、授权安全、数据安全 |
| 12. 可用性覆盖 | 易用性、可访问性、国际化、帮助文档 |

### 6.2 优先级分布要求

| 等级 | 占比 | 适用范围 |
|------|------|----------|
| P1 🔥 | 10%-15% | 仅限核心业务流程正向执行场景，功能失效会导致系统核心价值无法实现 |
| P2 ⚖️ | 60%-70% | 重要业务功能的正常和异常场景、核心功能边界值测试、数据验证和格式检查 |
| P3 📝 | 20%-25% | 辅助功能、管理后台功能、边缘场景、兼容性测试、UI交互优化验证 |

### 6.3 覆盖率要求

- **核心功能**：100%覆盖
- **重要功能**：95%覆盖
- **一般功能**：80%覆盖
- **辅助功能**：60%覆盖

## 七、测试用例生成规则

### 7.1 质量要求标准

1. **业务场景真实性**：每个测试用例必须基于文档中的真实业务场景
2. **测试步骤可执行性**：测试步骤必须详细具体，测试人员可直接按步骤操作
3. **预期结果明确性**：预期结果必须具体明确，便于判断测试是否通过
4. **无重复模板化**：每个测试用例都必须有独特的测试目标，严禁模板化重复

### 7.2 测试场景强制识别机制

**核心原则：场景驱动，而非数量驱动**

#### 第一步：深度分析文档（必须）
在生成每个模块的测试用例前，必须：
1. 仔细阅读该模块对应的文档章节（从 start_line 到 end_line）
2. 识别文档中的所有功能点、交互点、数据点
3. 提取具体的业务规则、验证规则、状态规则
4. 不遗漏任何测试场景

#### 第二步：输出测试场景清单（强制）
必须输出格式化的场景清单，包含：
- 功能正常流程场景：2-4个（来自文档的业务流程描述）
- UI交互场景：2-3个（来自文档的按钮、弹窗、跳转描述）
- 数据验证场景：2-3个（来自文档的字段、格式、长度限制）
- 异常处理场景：2个（来自文档的异常处理或合理推断）
- 边界值场景：1-2个（来自文档的数值限制）
- 兼容性场景：1-2个（基于产品特性）

每个场景必须：
- 描述具体（不是"功能测试"这种泛泛的）
- 注明来源（文档第X行或合理推断）
- 有明确的测试目标

#### 第三步：场景覆盖检查（强制）
场景清单输出后，必须自检：
- 是否覆盖了12个测试维度？
- 场景总数是否≥11个？
- 如果<11个 → 重新分析文档，补充遗漏场景

#### 第四步：基于场景生成用例（一场景一用例）

#### 用例数量说明
**每个模块11-14个用例是全面覆盖的自然结果**，分布参考：
- 功能正常流程测试：2-4个
- 数据验证测试：2-3个
- UI交互测试：2-3个
- 异常场景测试：2个
- 边界值测试：1-2个
- 兼容性测试：1-2个

### 7.3 测试用例 JSON 格式

```json
{
  "测试模块": "具体的功能模块名称",
  "用例名称": "描述具体测试场景的简洁标题",
  "前置条件": "具体的环境要求和数据准备",
  "用例步骤": "1. 第一步操作\n2. 第二步操作\n3. 第三步操作",
  "预期结果": "明确的期望行为和结果状态",
  "用例等级": "P1/P2/P3",
  "开发验证人员": "",
  "冒烟测试结果": "",
  "测试人员": "",
  "测试结果-安卓": "",
  "测试结果-iOS": "",
  "需求原文": "[章节名] 第X行\n具体的需求原文内容",
  "备注": "测试要点和注意事项"
}
```

### 7.4 需求原文提取规则

#### 提取标准

| 类型 | 示例 | 说明 |
|------|------|------|
| 功能描述句 | "点击加购icon，直接将该商品加入购物车" | 直接描述功能行为的语句 |
| 业务规则 | "先取即将发售的卡包，不足3个再取已发售卡包补足" | 明确的业务逻辑规则 |
| 数据限制 | "最长20个汉字"、"最大5M" | 字段限制、边界值 |
| 计算公式 | "金额 = 单价 × 数量 - 折扣" | 数学公式或计算逻辑 |
| 状态描述 | "未发售显示「距离发售还有X天！」" | 状态转换或显示规则 |
| UI交互要求 | "横幅关闭后，对该用户不再展示" | 界面交互行为描述 |

#### 提取原则
- **保持原文完整性**：直接引用文档原文，不做修改或改写
- **精准定位**：提取与该测试用例直接相关的最小语义单元
- **多条合并**：一个用例对应多条需求时，用分号（；）分隔
- **格式清理**：移除 Markdown 格式符号，保留纯文本

#### 定位信息格式
- **第一行**：`[章节名] 第X行`
- **第二行**：原文内容（换行显示）
- 行号为章节内相对行号（从章节标题下一行开始计为第1行）

#### 允许为空的情况
- 异常场景测试（如网络错误、服务器异常）
- 兼容性测试（如安卓/iOS适配）
- AI推导的边界值测试（文档未明确描述）
- 安全性测试（如SQL注入、XSS）
- 探索性测试（文档未明确描述的场景）

### 7.5 严禁事项

- 禁止使用相同的测试步骤模板
- 禁止使用模糊的预期结果描述
- 禁止使用"xxx"等占位符
- 禁止为不同模块生成相同的测试用例
- 禁止添加文档中未提及的功能
- 禁止忽略文档中的具体要求
- 禁止使用与业务场景不符的测试数据
- 禁止生成无法执行的测试步骤

## 八、测试用例内容质量规则

### 8.1 用例名称生成规则

#### 直接从业务功能提取核心名称：
- 识别文档中的核心业务动作（如：登录、注册、支付、查询等）
- 提取关键业务对象（如：订单、用户、商品、邮件等）
- 组合形成简洁的功能描述

#### 避免使用原始文档格式：
- 不直接复制包含 Markdown 格式的文本
- 不使用表格格式符号
- 不包含图片引用
- 不使用 HTML 标签和属性

#### 标准格式：
- **功能测试**：[业务对象][业务动作]功能测试
- **场景测试**：[业务场景]完整流程测试
- **异常测试**：[异常情况]异常处理测试
- **边界测试**：[数据项][边界类型]边界测试-[具体情况]
  - 必须基于文档中的具体数字限制生成
  - 每个限制至少生成3个测试用例：小于限制、等于限制、大于限制

### 8.2 测试步骤优化规则

#### 操作具体化：
- 不使用"执行XXX操作"等模糊表述
- 提供具体的操作步骤
- 包含明确的界面元素定位

#### 步骤可执行性：
- 每个步骤都应该是可操作的具体动作
- 包含必要的输入数据和参数
- 提供清晰的操作路径

#### 逻辑连贯性：
- 步骤之间保持逻辑连贯
- 前后步骤有明确的依赖关系
- 符合用户实际操作流程

### 8.3 预期结果优化规则

#### 结果可验证性：
- 避免直接复制用例名称作为预期结果
- 提供具体的验证标准
- 包含可观察的现象描述

#### 多层次验证：
- **即时验证**：操作完成后的即时反馈
- **数据验证**：数据库中数据的正确性
- **集成验证**：与其他系统的集成效果
- **业务验证**：业务规则的正确执行

### 8.4 质量检查机制

#### 语义完整性检查：
- 确保名称表达完整的测试意图
- 避免语义不通或不完整的表述
- 保证中文语法正确

#### 业务准确性检查：
